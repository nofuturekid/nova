# ADR-NNNN: Short title (imperative mood)

- **Status**: Proposed | Accepted | Deprecated | Superseded by ADR-XXXX
- **Date**: YYYY-MM-DD
- **Tags**: ci, release, ui, data, security, …

## Context

What problem are we solving? What forces are at play (technical, social, organisational, regulatory)? What constraints exist? Keep it factual — describe the situation, not the decision yet.

## Decision

State the decision in active voice, one or two sentences. Then expand on the *what* — the concrete approach, the shape of the change, the API or file boundaries it touches.

## Consequences

What changes as a result? Cover both the good and the bad:

- **Positive** — what this unlocks or simplifies.
- **Negative / trade-offs** — what it costs, complicates, or constrains.
- **Trigger to revisit** — concrete condition that would invalidate this decision (e.g. "if we add a third caller", "if the team grows past N people", "if the dependency reaches version X"). Without this an ADR rots silently.

## Alternatives considered

For each alternative, one paragraph: what it was, why we didn't pick it. Avoid strawmen — represent the alternatives charitably.
